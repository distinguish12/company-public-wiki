# SOP-2026-001 - 新员工入职知识库阅读与提交指南

> **文档定位**：本文档为新员工入职必读流程规范，旨在确保每位新成员在入职初期快速了解公司知识体系，并通过标准化的 PR 流程完成阅读确认。
> **目标读者**：所有新入职员工。
> **负责人**：@zhaopeng

---

## 一、目的

1. 帮助新员工快速融入公司的知识体系与研发文化。
2. 通过"阅读 → 输出 → 提交"的闭环，确保知识真正被理解与吸收，而非走过场。
3. 所有阅读确认记录以 Git 历史留存，可追溯、可审计。

---

## 二、必读知识库清单

新员工须在**入职首周内**完成以下知识库的通读：

| 序号 | 知识库 / 文档 | 重点关注内容 |
| :---: | :--- | :--- |
| 1 | `company-public-wiki` — 本仓库全部文档 | 公司战略目标、制度、通用 SOP |

> **💡 提示**：根据岗位不同，直属负责人可额外指定 `business-operations`（运营库）或 `infrastructure-maintenance`（运维库）作为补充阅读材料。

---

## 三、阅读完成后的提交要求

### 3.1 提交什么

阅读完成后，新员工须撰写一份 **「入职阅读总结」** 文档，内容包含：

1. **对公司知识体系的整体理解**：用自己的话概述公司的知识架构与协同方式。
2. **印象最深的 3 个知识点**：说明为什么印象深刻，以及对自己工作的启发。
3. **疑问与建议**：对知识库内容的任何疑问，或对现有文档的改进建议。

### 3.2 文档格式规范

提交的文档**必须**遵循公司全域统一的 Front Matter Schema（参见 [R09 §4.1 全域统一的基础 Schema](R09-AI研发与知识协同体系全景指南.md)）：

| 字段名 | 必填 | 说明/允许的枚举值 |
| :--- | :---: | :--- |
| `id` | 是 | 唯一标识。如 `FAM-2026-001`, `LEARN-2026-042`, `SPEC-2026-005` |
| `type` | 是 | `spec` (需求), `adr` (决策), `sop` (流程), `policy` (制度), `learning` (学习), `diary` (日记), `finance` (财务) |
| `confidence_level` | 是 | **置信度 (矛盾治理核心)**：`draft`, `hypothesis`, `verified`, `deprecated`, `obsolete` |
| `context_boundary` | 否 | 适用边界。描述知识成立的前提，如 "适用于5-10人初创期", "Java高并发场景" |
| `superseded_by` | 否 | 替代者 ID。当 `confidence_level` 降级为 `deprecated` 时必填 |
| `owner` | 是 | 负责人 GitHub Handle (如 `@zhaopeng`) |
| `sync_target` | 否 | 推送到外部系统的目标，如 `["notion", "dingtalk_group_a"]` |
| `tasks` | 否 | (仅限 `type: spec` 时) 包含子任务 `title`, `repo`, `assignee` 等信息，用于自动跨库派发 |

**文档模板示例**：

```yaml

type: learning

context_boundary: "新员工入职阅读总结"
```

```markdown

1. **知识点一**：……（为什么印象深刻？对你的启发是什么？）

- （对知识库内容的疑问）

```

### 3.3 提交到哪里

所有新员工的阅读总结统一存放在本仓库的固定目录下：

```
company-public-wiki/
└── onboarding/          ← 新员工阅读总结存放区域
    ├── ONBOARD-2026-001-张三.md
    ├── ONBOARD-2026-002-李四.md
    └── ...
```

### 3.4 如何提交 (PR 流程)

请严格按照以下步骤提交，**禁止直接 push 到 main 分支**：

1. **Fork 或 Clone 本仓库**，并切换到新分支：
   ```bash
   git checkout -b onboarding/你的姓名
   ```

2. **在 `onboarding/` 目录下创建你的阅读总结文档**：
   ```bash
   # 文件命名格式：ONBOARD-2026-XXX-姓名.md
   # 示例：
   touch onboarding/ONBOARD-2026-001-张三.md
   ```

3. **编写文档内容**（参照上方模板，确保 Front Matter 完整）。

4. **提交并推送**：
   ```bash
   git add onboarding/
   git commit -m "docs(onboarding): 新员工阅读总结 - 你的姓名"
   git push origin onboarding/你的姓名
   ```

5. **在 GitHub 上发起 Pull Request**：
   - Base 分支选择 `main`
   - 标题格式：`[Onboarding] 新员工阅读总结 — 你的姓名`
   - 在 PR 描述中简要说明已完成阅读的知识库清单

6. **等待 Review 与合并**：
   - 你的直属负责人或 @zhaopeng 将对 PR 进行 Review
   - 如有修改建议，请在分支上继续修改并推送
   - Review 通过后由负责人合并至 `main`

## 五、常见问题 (FAQ)

**Q1：我不熟悉 Git 和 PR 操作怎么办？**
A：这正是入职学习的一部分。请先阅读 `R09` 中关于 PR 工作流的章节，或向团队成员寻求帮助。掌握 Git 协同是每位员工的基本技能。

**Q2：阅读总结写多长合适？**
A：不限字数，重质量不重数量。能清晰表达你的理解和思考即可，建议 500-1500 字。

**Q3：提交后还能修改吗？**
A：可以。在 PR 合并前，你可以随时在分支上推送新的 commit 来修改内容。合并后如需修改，请发起新的 PR。

**Q4：我的总结中包含对现有文档的改进建议，会被采纳吗？**
A：非常欢迎！好的建议会被转化为独立的 Issue 进行跟踪落地。新人视角往往能发现老员工习以为常的问题。
