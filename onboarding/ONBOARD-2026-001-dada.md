# 入职阅读总结 — dada

## 一、对公司知识体系的整体理解

通过阅读 `company-public-wiki` 仓库的文档，我对公司的知识管理体系有了以下理解：

### 知识架构设计

公司采用了 **"文档即代码"（Docs-as-Code）** 的知识管理理念，所有知识文档都存储在 Git 仓库中，通过 PR 流程进行审核和更新。这种方式具有以下优势：

1. **版本可追溯**：每一次修改都有 Git 历史记录
2. **协作透明**：通过 PR 机制确保内容质量
3. **自动化集成**：利用 GitHub Actions 实现智能审批

### Front Matter Schema 设计理念

公司定义了统一的前置元数据 Schema，核心字段包括：

- `id`：唯一标识，便于引用和管理
- `type`：文档类型分类（需求、决策、流程、制度、学习、日记、财务）
- `confidence_level`：置信度，这是**矛盾治理的核心**，用于标记知识的可靠性
- `context_boundary`：适用边界，明确知识成立的前提条件

这种设计解决了知识管理中的 **"知识过期"和"上下文缺失"** 两大痛点。

### 协同方式

```
┌─────────────┐    PR     ┌─────────────┐   Auto Merge   ┌─────────────┐
│  新员工提交  │ ────────▶ │  AI 审核    │ ─────────────▶ │   main 分支  │
│  阅读总结    │           │  (安全类型)  │                │             │
└─────────────┘           └─────────────┘                └─────────────┘
```

## 二、印象最深的 3 个知识点

### 1. 置信度驱动的知识治理

**知识点**：`confidence_level` 字段的设计 — `draft → hypothesis → verified → deprecated → obsolete`

**为什么印象深刻**：
这是一个非常巧妙的设计。大多数知识库只关注知识的创建，却忽略了知识的生命周期。通过置信度字段，可以：
- 明确标记知识的可靠程度
- 当知识过期时，通过 `superseded_by` 字段指向新版本
- 避免团队依赖过时或错误的知识

**启发**：在未来的工作中，我会特别注意文档的置信度，避免将 `hypothesis` 阶段的知识当作已验证的事实使用。

### 2. Front Matter 驱动的自动化审批

**知识点**：GitHub Actions 根据 `type` 字段自动判断是否合并 PR

**为什么印象深刻**：
这是一个将 **"约定优于配置"** 发挥到极致的例子：
- `type: learning` → 自动合并（低风险）
- `type: finance` + amount ≤ 500 → 自动合并（小额报销）
- `type: finance` + amount > 500 → 人工审批（大额财务）

**启发**：这种设计思想可以推广到其他场景，比如：
- 自动生成文档目录
- 根据 `sync_target` 推送到不同平台
- 根据 `tasks` 自动派发工单

### 3. 新员工入职的"阅读→输出→提交"闭环

**知识点**：SOP 中要求新员工必须撰写阅读总结并通过 PR 提交

**为什么印象深刻**：
这个流程设计非常科学：
1. **被动阅读**容易流于形式
2. **主动输出**强迫新员工思考和消化
3. **PR 提交**让审核者能看到新员工的理解程度
4. **Git 记录**保留入职过程的完整轨迹

**启发**：这种"输入-输出-反馈"的闭环模式，比单纯的"阅读确认"有效得多。我会在后续的工作中也采用类似的方式来确保知识传递的有效性。

## 三、疑问与建议

### 疑问

1. **关于 `sync_target` 的具体实现**：
   - 文档中提到 `sync_target: ["notion", "dingtalk_group_a"]`，但目前是否已有对应的同步机制？
   - 如果有，是实时同步还是定时同步？

2. **关于置信度的维护机制**：
   - 当知识更新时，是否有自动提醒 owner 更新 `confidence_level` 的机制？
   - 是否有定期 review 已发布知识可靠性的流程？

### 建议

1. **增加文档模板的 IDE 支持**：
   建议提供 VS Code snippet 或 JSON Schema，让新员工在编辑 Front Matter 时有智能提示。

2. **完善 auto-merge 的通知机制**：
   当 PR 被 AI 自动合并后，可以增加一条评论说明：
   ```
   🤖 此 PR 已根据 Front Matter 规则自动合并。
   如有疑问，请联系 @zhaopeng
   ```

3. **考虑增加 `reviewers` 字段**：
   在 Front Matter 中支持 `reviewers: ["@xxx", "@yyy"]`，这样特定类型的文档可以自动指派审核者。
